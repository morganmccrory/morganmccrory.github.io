Version Control: Git and GitHub

Version control monitors the changes that you make to a file. It is a safe-guard against mistakes – and let’s be honest, we all make them. Have you ever written an email, sent it without properly proofreading, and then discovered your error only after it was sent? Or maybe you did proofread, but somehow one misspelling just slipped by you. That is where version control comes in!



It remembers every single change you make in a code, no matter how big or small. If you make an error without realizing it, and change a huge chunk of code in the meantime, do not fear! You still have a previous version readily available that you can cleanly work from. It really is a life-saver for developers.



Yet, it isn’t only limited to developers! No, no – just about anyone can use version control. An accountant can use it to keep track of financial transactions, and a writer can use it to keep track of versions of a story. Each commit could be another chapter, revisions to a previous chapter, or even changes to a character’s name. Fun fact: In the Harry Potter series, Hermione Granger’s original last name was Puckle, but J.K. Rowling felt that “the name did not suit her at all.” Version control might have made this important revision much easier.



There are lots of different version control systems to use, but the most popular ones for developers are Subversion (SVN) and Git. They each have their own strengths and weaknesses, but I am partial to using Git, so that is the one that we will talk about here.

I Don’t “Git” It

Git, which is a type of version control, tracks your changes by keeping track of all of your "commits." A commit is the your official commitment to making a change. It is a "snapshot" of your code that you can reference at any time. Usually, commits are pushed after every small change. That way, instead of sending in many updates at once, only a small update is sent. Git monitors all of these commits so you can see the process that you went through to create the final version of a program. There is a popular phrase used in the git community: Commit early, commit often! It won’t hurt anything to have a long list of commits. However, if you change too much at once, you might not be able to go back to the version that you wanted and this will result in lots of wasted time. There are some standard practices to use when writing commit messages, and you can read more about them here.

Let’s Work Together

It seems obvious at this point that version control is extremely beneficial for keeping track of your own files, but you may not have even considered how important is it when working on the same files with other collaborators. Back in my day, when working on a group paper in school, we would have to email the paper back and forth, each one emailing the new version as they made edits. Now, we can all work on the same files at the same time. And if I don’t like an edit that you pushed through? That’s OK! We still have each of the previous versions saved.



Working on the same file at the same time is made possible by a process called merging. Merging takes commits by two or more people and combines them into a brand new file. You can work on the beginning of a paper while I work on the ending and we won’t overwrite each other! Pretty neat!



Git lets you create and work on what is called ”branches”. Essentially, branches are different versions of the same thing that you can switch back and forth out of interchangeably. For example, let’s pretend that you have five different versions of yourself. You have one version of you that is a sports star, another that is a football player, one that is a janitor, one that is an astronaut, and a last one that is a programmer. They’re all you – but just with slight modifications. Now you get to see how each one looks and then you can say, “You know what, programming is awesome and I want to keep the one where I am a programmer!” Then you can throw the other versions of yourself away and only keep that one. That is how branches work – you can see how they all look, but then decide which version you like better and “keep” that one.

GitHub

GitHub is used to store code – it lives on the internet. This is the website (ie. hub) that programmers can go to in order to share their code with other programmers and collaborate together online. Again, going back to emailing a school paper back and forth to your group partners, this takes away that step and let’s everyone contribute to one master version at the same time.



In order to use GitHub, you have to ”push” your changes to GitHub. They don’t just appear automatically if you’re working locally with Git. After installing git, and going through some initial processes to link up your GitHub account and your computer, actually changing files on GitHub is fairly simple! From your terminal, type:

git add /put/file/name/here
git commit –m “Add commit message here”
git push


And voila! Your changes have now been pushed to Github. What you do locally on your computer doesn’t automatically appear online – you need to explicitly send them there. I hope this has been a beneficial introduction!
